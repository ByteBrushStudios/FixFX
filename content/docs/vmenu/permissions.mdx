## Overview

vMenu uses FiveM's built-in **ACE (Access Control Entry)** and **Principal** system for permissions. This powerful system allows granular control over who can access specific features.

<Callout type="warn">
  **Critical Security**: By default, vMenu may grant all permissions to everyone. You MUST configure permissions before making your server public.
</Callout>

## Permission System Basics

### Key Concepts

| Term | Description |
|------|-------------|
| **ACE** | Access Control Entry - grants or denies a specific permission |
| **Principal** | An entity that can have ACEs (player, group, identifier) |
| **Group** | Collection of principals with shared permissions |
| **Identifier** | Unique player identifier (Steam, License, Discord, etc.) |

### How It Works

1. **Players** have **identifiers** (steam:, license:, discord:, etc.)
2. **Identifiers** can be added to **groups** (admin, moderator, vip)
3. **Groups** (or individual identifiers) receive **ACE permissions**
4. vMenu checks permissions when players access features

## Permission Configuration

### Method 1: Using permissions.cfg (Recommended)

Located at `resources/vMenu/config/permissions.cfg`

#### Basic Structure

```cfg
# ===================================
# vMenu Permissions Configuration
# ===================================

# ----- ADMIN GROUP -----
add_ace group.admin vMenu.Everything allow

# Assign players to admin group
add_principal identifier.steam:110000112345678 group.admin
add_principal identifier.license:abc123def456 group.admin

# ----- MODERATOR GROUP -----
add_ace group.moderator vMenu.OnlinePlayers allow
add_ace group.moderator vMenu.PlayerOptions allow
add_ace group.moderator vMenu.Report allow
add_ace group.moderator vMenu.Spectate allow
add_ace group.moderator vMenu.Teleport allow

# Assign moderators
add_principal identifier.steam:110000187654321 group.moderator
add_principal identifier.discord:123456789012345678 group.moderator

# ----- VIP GROUP -----
add_ace group.vip vMenu.PersonalVehicle allow
add_ace group.vip vMenu.SavedVehicles allow
add_ace group.vip vMenu.WeaponLoadouts allow
add_ace group.vip vMenu.SavedCharacters allow

# Assign VIP members
add_principal identifier.license:vip123 group.vip

# ----- DEFAULT PERMISSIONS -----
# Deny dangerous permissions by default
add_ace builtin.everyone vMenu.Everything deny
add_ace builtin.everyone vMenu.Staff deny
add_ace builtin.everyone vMenu.NoClip deny
add_ace builtin.everyone vMenu.OnlinePlayers.All deny
```

### Method 2: Using server.cfg

Add permissions directly in your `server.cfg`:

```cfg
# Admin permissions
add_ace group.admin vMenu.Everything allow
add_principal identifier.steam:110000112345678 group.admin

# Moderator permissions
add_ace group.moderator vMenu.OnlinePlayers allow
add_principal identifier.steam:110000187654321 group.moderator
```

## Finding Player Identifiers

### In-Game Method

1. Player joins server
2. Check server console for connection message:
   ```
   [  script:vMenu] Player connected: PlayerName
   [  script:vMenu] steam:110000112345678
   [  script:vMenu] license:abc123def456ghi789
   [  script:vMenu] discord:123456789012345678
   ```

### Console Command

Use the `status` command in server console:

```
status
```

Output:
```
id: 1, PlayerName, steam: 110000112345678, license: abc123
```

### Script Method

Use this snippet to log identifiers:

```lua
-- Add to a resource
RegisterCommand('myid', function(source)
    local identifiers = GetPlayerIdentifiers(source)
    for _, id in pairs(identifiers) do
        print(id)
    end
end, false)
```

## Permission Hierarchy

### Understanding Allow/Deny

```cfg
# Deny takes precedence over allow
add_ace builtin.everyone vMenu.Everything deny  # Denies everyone
add_ace group.admin vMenu.Everything allow      # Allows admins only
```

### Group Inheritance

Groups don't inherit from each other by default. Use principals:

```cfg
# Wrong - groups don't inherit
add_ace group.admin vMenu.Everything allow
add_principal group.moderator group.admin  # This doesn't work as expected

# Right - explicitly add player to both groups
add_principal identifier.steam:110000112345678 group.admin
add_principal identifier.steam:110000112345678 group.moderator
```

### Permission Specificity

More specific permissions override general ones:

```cfg
# Allow all vehicle spawning
add_ace group.vip vMenu.VehicleSpawner allow

# But deny military vehicles
add_ace group.vip vMenu.VehicleSpawner.Military deny
```

## Advanced Permission Patterns

### Time-Limited Permissions

Use external script with ACE system:

```lua
-- Grant VIP for 30 days
RegisterCommand('grantvip', function(source, args)
    local target = tonumber(args[1])
    local identifier = GetPlayerIdentifier(target, 0)
    
    -- Add to VIP group
    ExecuteCommand(string.format('add_principal identifier.%s group.vip', identifier))
    
    -- Schedule removal (requires database/scheduler)
    -- Implementation depends on your framework
end, true)
```

### Dynamic Permission Assignment

Integrate with your framework:

```lua
-- ESX Example
RegisterServerEvent('esx:playerLoaded')
AddEventHandler('esx:playerLoaded', function(source, xPlayer)
    if xPlayer.getGroup() == 'admin' then
        ExecuteCommand('add_principal identifier.'..xPlayer.identifier..' group.admin')
    end
end)
```

### Conditional Permissions

Based on player stats:

```lua
-- Grant features based on playtime
AddEventHandler('playerConnecting', function()
    local source = source
    local playtime = GetPlayerPlaytime(source) -- Your function
    
    if playtime > 100 then  -- 100 hours
        local identifier = GetPlayerIdentifier(source, 0)
        ExecuteCommand('add_principal identifier.'..identifier..' group.veteran')
    end
end)
```

## Security Best Practices

### Essential Security Measures

1. ‚úÖ **Always deny dangerous permissions by default**
   ```cfg
   add_ace builtin.everyone vMenu.Everything deny
   add_ace builtin.everyone vMenu.NoClip deny
   add_ace builtin.everyone vMenu.Staff deny
   ```

2. ‚úÖ **Use specific permissions instead of wildcards**
   ```cfg
   # Bad - too permissive
   add_ace group.moderator vMenu.OnlinePlayers.All allow
   
   # Good - specific permissions
   add_ace group.moderator vMenu.OnlinePlayers.Kick allow
   add_ace group.moderator vMenu.OnlinePlayers.TempBan allow
   ```

3. ‚úÖ **Use identifier-based permissions, not IP-based**
   ```cfg
   # Bad - IPs change
   add_principal identifier.ip:127.0.0.1 group.admin
   
   # Good - stable identifiers
   add_principal identifier.steam:110000112345678 group.admin
   ```

4. ‚úÖ **Regular audit of permissions**
   - Review permissions monthly
   - Remove inactive admins
   - Check for permission conflicts

5. ‚úÖ **Log permission usage**
   
   Check the [official vMenu documentation](https://docs.vespura.com/vmenu/configuration) for logging options.

### Common Security Mistakes

‚ùå **Granting Everything to everyone**
```cfg
# Never do this!
add_ace builtin.everyone vMenu.Everything allow
```

‚ùå **Using weak identifier types**
```cfg
# IP addresses change frequently
add_principal identifier.ip:192.168.1.1 group.admin
```

‚ùå **Not denying dangerous permissions**
```cfg
# Forgetting to explicitly deny
# (some features may be allowed by default)
```

‚ùå **Sharing admin accounts**
```cfg
# Each admin should have unique identifier
add_principal identifier.license:shared_account group.admin
```

## Troubleshooting Permissions

### Permission Not Working

**Issue**: Player has permission but can't access feature

**Solutions**:
1. Verify permission syntax is exact (case-sensitive)
2. Check for deny overriding allow
3. Restart server after permission changes
4. Verify player identifier is correct
5. Check if feature is disabled via convar

### Too Many Permissions

**Issue**: Player has access to features they shouldn't

**Solutions**:
1. Check if accidentally added to admin group
2. Review `builtin.everyone` permissions
3. Check for `vMenu.Everything` or wildcard permissions
4. Verify no conflicting permission systems

### Permissions Not Persistent

**Issue**: Permissions reset after restart

**Solutions**:
1. Add permissions to `permissions.cfg` or `server.cfg`
2. Don't use console commands for permanent permissions
3. Verify files are being read on startup
4. Check file permissions (read/write access)
